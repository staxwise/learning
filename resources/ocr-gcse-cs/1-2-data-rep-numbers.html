01 · BINARY

The Binary Number System

Binary is base 2 — the only digits are 0 and 1. Each position (column) has a place value that is double the previous one, giving the sequence: 128 · 64 · 32 · 16 · 8 · 4 · 2 · 1.

CLICK EACH BUTTON TO REVEAL ▼

PLACE VALUES
Bit position8 (MSB)7654321 (LSB)
Place value1286432168421
Power of 22⁷2⁶2⁵2⁴2³2²2¹2⁰
  • MSB (Most Significant Bit) = leftmost bit = highest place value (128). Changing it has the biggest impact on the number.
  • LSB (Least Significant Bit) = rightmost bit = lowest place value (1). Changing it has the smallest impact.
  • The spec requires you to work with numbers containing between 1 and 8 bits — e.g. 11010 is the same as 00011010 (pad with leading zeros to make 8 bits).
BINARY TO DENARY

Write the place values above each bit. For every bit that is 1, add its place value. Bits that are 0 contribute nothing.

Worked Example — Convert 10110100 to denary
1
Write the place values: 128 | 64 | 32 | 16 | 8 | 4 | 2 | 1
2
Write the binary number underneath:  1  |  0 |  1 |  1 |  0 |  1 |  0 |  0
3
Identify which bits are 1: positions 128, 32, 16, 4
4
Add those place values: 128 + 32 + 16 + 4 = 180
✓
10110100 in binary = 180 in denary
DENARY TO BINARY

Work from the largest place value down to 1. Ask: "Does this place value fit into my remaining number?" If yes, write 1 and subtract it; if no, write 0.

Worked Example — Convert 217 to binary
1
128 fits into 217? Yes (217 − 128 = 89) → write 1
2
64 fits into 89? Yes (89 − 64 = 25) → write 1
3
32 fits into 25? No → write 0
4
16 fits into 25? Yes (25 − 16 = 9) → write 1
5
8 fits into 9? Yes (9 − 8 = 1) → write 1
6
4 fits into 1? No → write 0. 2 fits into 1? No → write 0
7
1 fits into 1? Yes (1 − 1 = 0) → write 1
✓
217 in denary = 11011001 in binary

  • MSB = leftmost bit (value 128). LSB = rightmost bit (value 1)

02 · ADDITION

Binary Addition & Overflow

Adding binary numbers works exactly like denary addition — you work column by column from right to left, carrying when the sum exceeds the number of digits available.

CLICK EACH BUTTON TO REVEAL ▼

ADDITION RULES
ABSumCarryIn plain English
0000Zero plus zero is zero
0110Zero plus one is one
1010One plus zero is one
1101One plus one = 2 in denary = 10 in binary (write 0, carry 1)
11 + carry 1111+1+1 = 3 in denary = 11 in binary (write 1, carry 1)
💡 The Key Rule

In binary, 1 + 1 = 10 (not 2!). Read it as "zero, carry one". And 1 + 1 + 1 = 11. These are the two cases where a carry is generated.

STEP-BY-STEP WORKED EXAMPLE

Convert to check: 00110101 = 53, 01001110 = 78. Expected answer: 53 + 78 = 131 = 10000011

A (53)00110101
B (78)01001110
Result10000011

Result: 10000011 = 128 + 2 + 1 = 131 ✓

OVERFLOW

With 8 bits, the maximum value you can store is 255 (11111111). If the result of an addition exceeds 255, it doesn't fit — this is an overflow error.

Overflow Example — 11000000 + 11000000 (192 + 192 = 384)
1
192 + 192 = 384 — but 384 > 255, so it cannot fit in 8 bits
2
The result would be 110000000 — a 9-bit number
3
The 9th bit (the carry out of the MSB column) has nowhere to go — it is lost
⚠️
The 8-bit result stored is 10000000 = 128 — completely wrong! This is an overflow error.
⚠️ Exam Definition

Overflow occurs when the result of a binary calculation is too large to be stored in the available number of bits, causing the most significant carry bit to be lost and producing an incorrect result.

✓ CHECK — Q2
A computer adds two 8-bit values: 11100000 (224) and 00110000 (48). The result is 256. What happens?
AThe computer automatically increases its storage to 9 bits to accommodate the larger result
BThe result wraps around and is stored as 255, the largest 8-bit value
CAn overflow error occurs — the carry bit from the MSB column is lost, and the 8-bit result stored is 00000000 (0), which is incorrect
DThe operation is rejected and neither number is stored, because the sum exceeds the maximum byte value
✅ Correct! 224 + 48 = 256 = 100000000 in binary — that's a 9-bit result. The 9th bit (the leading 1) is lost, leaving 00000000. An overflow error has occurred: the stored result (0) is completely incorrect.
❌ 224 + 48 = 256 = 100000000 (9 bits). With only 8 bits available, the leading 1 (the carry out of the MSB column) is lost. The remaining 8 bits are 00000000. This is an overflow error — the stored result (0) is wrong.
  • Binary addition rules: 0+0=0, 0+1=1, 1+1=0 carry 1, 1+1+1=1 carry 1
  • Work right to left, carrying into the next column when sum ≥ 2
  • Maximum 8-bit value = 255 (11111111). Range is 0–255
  • Overflow = result exceeds 8 bits; the carry out of the MSB is lost; the stored result is wrong
  • Always check by converting to denary first — if sum > 255, overflow will occur
03 · HEX

Hexadecimal

Hexadecimal is base 16. Instead of reading long binary strings, computer scientists use hex as a compact shorthand — 4 bits collapse into just 1 hex digit.

CLICK EACH BUTTON TO REVEAL ▼

HEX DIGITS

Base 16 needs 16 symbols. Digits 0–9 aren't enough on their own, so we borrow letters A–F:

Hex0123456789ABCDEF
Denary0123456789101112131415
Binary0000000100100011010001010110011110001001101010111100110111101111

The spec range for 2-digit hex is 00 to FF (= 0 to 255 in denary). That exactly matches the 8-bit binary range.

THE PURPOSE OF HEX

❌ Binary — Hard to Read

  • A memory address: 11001010111100001010000000000001
  • 32 digits — easy to misread, very error-prone
  • No compact representation

✅ Hex — Compact & Clear

  • Same address: CAF0A001
  • 8 characters — much shorter, less error-prone
  • Every 4 bits → 1 hex digit, exactly
  • Memory addresses — displayed in hex in debuggers and OS tools
  • Colour codes — HTML/CSS colours use 6-digit hex e.g. #FF5733
  • MAC addresses — network hardware identifiers e.g. A4:C3:F0:85:AC:2D
  • Error codes — Windows BSODs show hex error codes (e.g. 0x0000007B)
CONVERTING DENARY TO HEX (2-DIGIT)

With 2-digit hex: the left digit = how many 16s, the right digit = the remainder.

Worked Example — Convert 197 to hex
1
Divide by 16: 197 ÷ 16 = 12 remainder 5
2
Left hex digit = 12 in denary = C in hex
3
Right hex digit = remainder 5 = 5 in hex
✓
197 in denary = C5 in hex

3
Right digit 3 = 3 × 1 = 3

  • Hexadecimal = base 16. Digits: 0–9, then A=10, B=11, C=12, D=13, E=14, F=15
  • 2-digit hex range: 00 to FF = 0 to 255 in denary (same as 8-bit binary)
  • Hex → Denary: left digit × 16, right digit × 1. Add the results.
  • Denary → Hex: divide by 16. Quotient = left digit, remainder = right digit
  • Used for: memory addresses, HTML colour codes, MAC addresses, error codes
  • Reason: 4 binary bits = 1 hex digit — compact and unambiguous shorthand for binary
04 · CONVERSION

Binary ↔ Hexadecimal

The most important relationship in this topic: 4 binary bits = 1 hex digit, exactly. This makes binary↔hex conversion fast and mechanical — no arithmetic needed.

CLICK EACH BUTTON TO REVEAL ▼

WHY IT WORKS

4 bits can represent 2⁴ = 16 values (0000 to 1111). Hex has exactly 16 symbols (0–F). So every possible 4-bit combination maps to exactly one hex digit — and vice versa.

The Rule
8-bit binary = 2 hex digits Split the 8 bits into two groups of 4: LEFT 4 BITS → hex digit 1 (left) RIGHT 4 BITS → hex digit 2 (right)
✅ No Arithmetic Required

Converting between binary and hex never requires division or subtraction — just split into nibbles (groups of 4) and look up the hex value. This is much faster than going through denary.

BINARY TO HEX
Example 1 — Convert 10110111 to hex
1
Split into two nibbles: 1011 | 0111
2
Left nibble 1011: 8+2+1 = 11 = B
3
Right nibble 0111: 4+2+1 = 7 = 7
✓
10110111 = B7 in hex
Example 2 — Convert 11010 to hex (less than 8 bits)
1
Pad to 8 bits with leading zeros: 11010 → 00011010
2
Split: 0001 | 1010
3
0001 = 1 = 1. 1010 = 8+2 = 10 = A
✓
11010 = 1A in hex
HEX TO BINARY

The reverse: convert each hex digit independently into its 4-bit equivalent, then join them together.

Example — Convert 3F to binary
1
Left digit 3 → 4-bit binary: 0011
2
Right digit F (=15) → 4-bit binary: 1111
✓
3F = 00111111 in binary
💡 Always Use 4 Bits Per Digit

When expanding a hex digit, always write exactly 4 binary digits. For hex digit 3, write 0011 — not 11. Leaving out the leading zeros will merge nibbles incorrectly.

✓ CHECK — Q4
Convert the binary number 11001110 to hexadecimal, showing the nibble split.
ACD — split: 1100 = C (=12), but 1110 = E not D
BC13 — treating it as three groups rather than two nibbles of 4
CCE — left nibble 1100 = 12 = C; right nibble 1110 = 14 = E

✅ Correct! Split 11001110 into nibbles: 1100 | 1110. Left: 8+4=12=C. Right: 8+4+2=14=E. Result: CE. Always split from the left — the leftmost nibble gives the leftmost hex digit.
❌ Split 11001110 into two nibbles of 4: left = 1100, right = 1110. Left: 8+4+0+0 = 12 → C. Right: 8+4+2+0 = 14 → E. Answer = CE. Don't mix up nibble order — left nibble maps to left hex digit.
  • 4 binary bits = 1 hex digit (a "nibble"). 8 bits = 2 hex digits
  • Binary → Hex: split 8 bits into two groups of 4 (left nibble, right nibble). Convert each nibble to one hex digit
  • Hex → Binary: convert each hex digit independently to a 4-bit binary group. Join the groups
  • Always pad to 4 bits per nibble — write 0011 not 11 for hex digit 3
  • Numbers with fewer than 8 bits: pad with leading zeros first (11010 = 00011010)
  • No arithmetic needed — binary↔hex is a direct substitution, digit by digit
05 · SHIFTS

Binary Shifts

A binary shift moves all the bits in a number left or right by a given number of positions. It's the most efficient way to multiply or divide by powers of 2.

CLICK EACH BUTTON TO REVEAL ▼

LEFT SHIFT

Shifting all bits n positions to the left multiplies the value by 2ⁿ. Zeros fill in on the right. Bits that fall off the left end are lost (potential overflow).

Example — Left shift 00000110 (6) by 2 positions
1
Original: 0 0 0 0 0 1 1 0 = 6
2
Shift left by 2: each bit moves 2 positions left, two 0s fill from the right
3
Result: 0 0 0 1 1 0 0 0 = 24
✓
6 × 2² = 6 × 4 = 24 ✓
⚠️ Left Shift Overflow

If a 1-bit is shifted off the left end (past the MSB), it is lost — this is an overflow. E.g. shifting 11000000 (192) left by 1 gives 10000000 (128), not 384 — the leading 1 was lost.

RIGHT SHIFT

Shifting all bits n positions to the right divides the value by 2ⁿ (integer division — remainders are lost). Zeros fill in on the left.

Example — Right shift 01011000 (88) by 3 positions
1
Original: 0 1 0 1 1 0 0 0 = 88
2
Shift right by 3: each bit moves 3 positions right, three 0s fill from the left
3
Result: 0 0 0 0 1 0 1 1 = 11
✓
88 ÷ 2³ = 88 ÷ 8 = 11 ✓
💡 Right Shift and Remainders

Right shifts perform integer division. If the bits shifted off the right are not all zero, information is lost. E.g. right-shifting 00000111 (7) by 1 gives 00000011 (3), not 3.5 — the remainder is discarded.
