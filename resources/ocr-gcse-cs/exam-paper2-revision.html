🧠 SECTION 02

Computational Thinking

There are four key techniques — click each one to reveal the definition. Try to predict it first.

CLICK EACH BUTTON TO REVEAL ▼

DECOMPOSITION

Breaking a complex problem into smaller, more manageable sub-problems that can each be solved separately. Shown visually as a structure diagram — a tree with the main problem at the top, broken into modules below.

EXAM PHRASE: "break the problem into smaller parts / sub-problems"

ABSTRACTION

Removing or hiding unnecessary details to focus on only the aspects important for solving the problem. A map is a classic example — it abstracts real geography into simple symbols, hiding irrelevant detail.

EXAM PHRASE: "remove unnecessary detail / focus on what matters"

PATTERN RECOGNITION

Identifying similarities and repeated patterns within or between problems. Recognising patterns allows reuse of existing solutions — e.g. noticing that multiple parts of a program need the same validation, so you write one function and call it everywhere.

EXAM PHRASE: "identify similarities / reuse existing solutions"

ALGORITHMIC THINKING

Designing a clear, precise step-by-step set of instructions (an algorithm) that a computer can follow to solve the problem. This is the logical planning stage — before writing any actual code.

EXAM PHRASE: "a sequence of steps to solve the problem"

STRUCTURE DIAGRAM RULES
  • The top-level box is the whole system/problem
  • Each layer below represents a further level of decomposition
  • Lines connect parent modules to their children — never skip levels
  • Labels on boxes are the module or sub-program names
  • Examiners award marks for correct hierarchy AND correct labels
⚠️ EXAM TRAP — KNOW THE DIFFERENCE

Decomposition = structure (splitting into parts). Abstraction = detail (removing what you don't need).

In an exam, read the wording carefully — "breaking down" = decomposition; "hiding unnecessary detail" = abstraction.

Variables, Scope & Data Types

Click each topic to reveal the detail. Understanding scope is essential for Section B questions.

CLICK EACH BUTTON TO REVEAL ▼

GLOBAL VARIABLES

Declared outside all sub-programs — accessible from anywhere in the program, including inside functions and procedures. They persist throughout the program's lifetime.

  • Can be read and modified from any part of the code
  • Risk: any sub-program can accidentally change them
  • Use sparingly — only for values genuinely needed everywhere
LOCAL VARIABLES

Declared inside a sub-program (function or procedure) — only accessible within that sub-program. They are created when the sub-program is called and destroyed when it ends.

  • Parameters passed into a sub-program are also local variables
  • Cannot be accessed from outside the sub-program — prevents accidental modification
  • Better for maintainability — each sub-program is self-contained

  • 🌍 Global: accessible everywhere · can be changed anywhere (risky)
  • 📦 Local: only inside its sub-program · destroyed when sub-program ends
  • Parameters passed in = local variables in that sub-program
  • Boolean = True/False · Integer = whole · Real = decimal

TRACE TABLE METHOD
  • Create a column for every variable in the algorithm, plus an Output column
  • Work through line by line — one row per change
  • Only update the column for the variable that changed on that line
  • For loops: trace every iteration, including updating the loop counter itself
  • Evaluate conditions first — which branch runs?
  • Leave cells blank if that variable did not change on that line

⚠️ ZERO INDEXING

String indices in OCR pseudocode start at 0, not 1.

"Tiger" → T=0, i=1, g=2, e=3, r=4

A loop for i = 0 to word.length - 1 runs 5 times for "Tiger" (i = 0, 1, 2, 3, 4). This is where most trace table marks are lost — always double-check index values.

Sorting & Searching Algorithms

Reveal each algorithm in turn. You need to describe, trace, and compare them.

CLICK EACH BUTTON TO REVEAL ▼

BINARY SEARCH

⭐ Prerequisite: the list MUST be sorted first. Binary search will produce wrong results on an unsorted list.

  • Find the midpoint of the list
  • Compare the target to the midpoint value
  • If equal → found ✓
  • If target < midpoint → search the left half
  • If target > midpoint → search the right half
  • Repeat until found or list exhausted
  • Very efficient: eliminates half the remaining items each step (O log n)
MERGE SORT — DIVIDE AND CONQUER

Split the list in half repeatedly until every sub-list has one element (a single element is trivially sorted). Then merge the sub-lists back together in sorted order.

-- Example: split phase for [8, 3, 1, 5]
[8, 3, 1, 5]
[8, 3]   [1, 5]
[8] [3]  [1] [5]
→ Merge: [3, 8]  [1, 5]
→ Merge: [1, 3, 5, 8]

In an exam, show every split and every merge step. One mark per correct stage.

BUBBLE SORT

Repeatedly compares adjacent elements and swaps them if they are in the wrong order. After each full pass, the largest unsorted value "bubbles" to the end.

  • After pass 1: the maximum value is in its final position at the end
  • After pass 2: the second largest is in its final position
  • Continue until no swaps occur in a full pass (list is sorted)
  • Less efficient than merge sort for large lists
COMPARISON
AlgorithmSorted first?MethodBest for
Binary Search✓ RequiredHalve search space each stepLarge sorted lists
Linear Search✗ Not neededCheck every element in orderSmall / unsorted lists
Bubble Sort✗ Not neededRepeatedly swap adjacent pairsNearly-sorted lists

After pass 2, the second largest will be correct, and so on. The smallest element does not necessarily reach the front in one pass.
✗ In bubble sort, adjacent elements are compared and swapped if out of order, causing the largest element to bubble all the way to the end in one pass. After the first full pass, only the maximum value (9) is guaranteed in its final position. The smallest does not necessarily reach the front in one pass.
  • 🔍 Binary search: MUST be sorted first — prerequisite!
  • 🔀 Merge sort: split until single elements → merge back sorted
  • 🫧 Bubble sort pass 1: LARGEST value moves to end
  • Show every step in exam — marks awarded per stage

Boolean Logic & Logic Gates

Reveal each gate, then the truth table method, then the evaluation technique.

CLICK EACH BUTTON TO REVEAL ▼

AND GATE

Output is 1 only if both inputs are 1. If either input is 0, output is 0.

0 AND 0 = 0  |  0 AND 1 = 0  |  1 AND 0 = 0  |  1 AND 1 = 1

Memory: AND = both must agree to output 1.

OR GATE

Output is 1 if at least one input is 1. Output is 0 only when both inputs are 0.

0 OR 0 = 0  |  0 OR 1 = 1  |  1 OR 0 = 1  |  1 OR 1 = 1

Memory: OR = at least one is enough.

NOT GATE

Inverts a single input. Takes one input and flips it.

NOT 0 = 1  |  NOT 1 = 0

Memory: NOT = opposite. Also written as a bar over the letter: Ā.

TRUTH TABLE METHOD

  • List all combinations of 0s and 1s systematically

Defensive Design, Testing & Test Data

Reveal each concept below. The authentication vs validation distinction is heavily examined.

CLICK EACH BUTTON TO REVEAL ▼

AUTHENTICATION — WHO ARE YOU?

Checking that the user is who they claim to be. It verifies identity. Authentication happens before access is granted.

  • Username and password login
  • PIN entry
  • Biometric scan (fingerprint, face ID)
  • Two-factor authentication (2FA)
INPUT VALIDATION — IS THE DATA OK?

Checking that data entered is reasonable, complete, and in the correct format. It verifies data quality. Validation happens during data entry.

  • Range check — is the value within allowed limits?
  • Presence check — has the field been filled in?
  • Type check — is it the correct data type?
  • Format check — does it match the expected pattern? (e.g. postcode)
⚠️ EXAM TRAP — KNOW THE DIFFERENCE

🔐 Authentication = who are you? (identity check) — happens before access is granted.

✅ Validation = is this data acceptable? (data check) — happens during data entry.

Example: A password check at login = authentication. Checking a password is ≥ 8 characters long = validation.

TEST DATA TYPES

✅ Normal

Valid and typical data — should be accepted.

  • Range 1–10: use 5
  • Name field: use "Alice"

🔢 Boundary

At or near the edge of the valid range — tests the limits.

  • Range 1–10: test 1, 10, 0, 11
  • Test just-inside AND just-outside

❌ Erroneous

Completely invalid — should be rejected.

  • Wrong type: "hello" for a number
  • Wildly out of range: -999
✓ CHECK — Q6

CNormal: 1985 · Boundary: 1900 (or 2010) · Erroneous: "abc" or -500
DNormal: 2005 · Boundary: 2005 · Erroneous: 1899
✓ Correct. Normal: 1985 is typical and valid. Boundary: 1900 and 2010 are the exact limits (test both, and ideally 1899 and 2011 just outside). Erroneous: "abc" is wrong type; -500 is wildly out of range. Option A has boundary and erroneous swapped — 1899 is out of range (erroneous), not boundary.
✗ Boundary = at or near the edge of the valid range (1900 or 2010). Erroneous = completely invalid (wrong type or wildly out of range). Option A has these swapped: 1899 is erroneous (out of range), not boundary. Option B's boundary of 1901 is just inside the range, not at the actual boundary (1900). Option D uses the same value for normal and boundary.
  • 🔐 Authentication = WHO are you? (identity) · ✅ Validation = is data OK?
  • ✅ Normal = typical valid · 🔢 Boundary = at the edge · ❌ Erroneous = completely invalid
  • Boundary: test the limit itself AND just outside it
  • Erroneous: wrong data type OR far out of range

Translators, Compilers & IDEs

High-level code must be translated into machine code before it can run. Reveal each translator type and the IDE tools.

CLICK EACH BUTTON TO REVEAL ▼

COMPILER
  • Translates the entire program at once before execution
  • Produces a standalone executable file — no translator needed to run it
  • Faster execution — machine code runs directly
  • Errors reported all at once after compilation
  • Source code not needed by the end user
  • Best for final / deployed versions of software
INTERPRETER
  • Translates and executes the program line by line
  • No standalone executable produced
  • Slower execution — translation happens at run time
  • Stops at the first error — easy to pinpoint during development
  • Requires the interpreter to be installed to run the code
  • Best for development and testing phases
IDE TOOLS
ToolWhat it does
Run-time environmentExecutes the program and lets you observe behaviour — see variables, outputs, and errors as it runs
Error diagnostics / debuggerHighlights errors, shows which line caused the problem, may suggest fixes
BreakpointsPauses execution at a specific line so you can inspect variable values at that moment
Syntax highlightingColours keywords, strings, variables differently — makes code easier to read and spot errors
Auto-completionSuggests completions as you type — speeds up development and reduces typing errors

DCompilers automatically fix syntax errors in the code before producing the executable
✓ Correct. For a deployed product, the compiler's key advantages are: (1) standalone executable — customers don't need the translator installed; (2) compiled machine code runs faster. Option C describes how an interpreter works (line by line), not a compiler. Option D is wrong — compilers report errors but do not fix them.

  • ⚙️ Compiler: whole program → executable · fast to run · errors all at once
  • 🔄 Interpreter: line by line · stops at 1st error · needs interpreter every time
  • Final product → compiler · Development → interpreter
  • 🛠 IDE: breakpoints (pause at line) · run-time env (observe execution)

❌ Avoid

  • Forgetting the output statement
  • Inverted conditions (most common logic error)
  • Off-by-one errors in loop bounds
  • Not re-reading input inside a do-while
  • Using = (assignment) when you mean == (comparison)

✓ CHECK — Q8
A while loop should keep asking for input until the user enters a valid positive number. The condition is: while number > 0. What type of error is this, and what is the effect?
ASyntax error — the while keyword has been used incorrectly and the program will not compile
BLogic error — the condition is inverted; the loop runs when the number IS positive (the valid case), so it immediately exits when valid data is entered
CRun-time error — the program will crash when the user enters 0
DLogic error — the loop will run infinitely because the condition can never become false
✓ Correct. This is a logic error — the program runs without crashing, but behaviour is wrong. while number > 0 means "keep looping while positive" — the exact opposite of the intent. The correct condition should be while number <= 0 (keep looping while the input is NOT valid). The program exits the loop immediately when valid input is entered — perfectly backwards.
✗ This is a logic error — inverted condition. The syntax is valid (no compile error) and there's no crash. The problem is purely logical: while number > 0 means "repeat while positive" — but positive is the valid case we want to ACCEPT, not loop on. The correct condition is while number <= 0. The program runs but does the opposite of what was intended.
  • Syntax = grammar broken · Logic = runs but wrong · Run-time = crashes
  • Logic errors: inverted condition, wrong operator (> vs >=), off-by-one
  • Always check: loop condition correct? Output statement present?
  • OCR ERL random: num = randint(1, 20)

2D ARRAYS — arrayName[row][col]

A 2D array is a grid — rows and columns stored as a single variable. Access elements with two indices: arrayName[row][col]. Both indices are 0-based.

-- Declare a 3x3 2D array:
array grid[3][3]

-- Assign a value to row 1, column 2:
grid[1][2] = "Hello"

-- Loop through ALL elements using nested loops:
for row = 0 to 2
    for col = 0 to 2
        print(grid[row][col])
    next col
next row

Outer loop = rows · Inner loop = columns. A 10×10 grid needs 100 total iterations.

FUNCTIONS vs PROCEDURES

⚙️ Function

  • Sub-program that returns a value
  • Uses the return keyword
  • Call: result = myFunc(arg)

📋 Procedure

  • Sub-program that performs a task
  • Does not return a value
  • Outputs directly or modifies globals
function CheckValue(row, col)
    if myArray[row][col] == "Empty" then
        return True
    else
        return False
    endif
endfunction

-- Calling the function:
result = CheckValue(3, 5)
MAINTAINABILITY TECHNIQUES
  • Comments — explain what sections do: # this counts empty seats
  • Meaningful names — totalSeats not t, playerScore not ps
  • Modularisation — break code into reusable functions/procedures
  • Indentation — consistent indentation makes structure clear
  • Constants — use named constants for values that don't change: MAX_SEATS = 100

⚠️ GLOBAL VARIABLES HARM MAINTAINABILITY

Making all variables global is a poor practice. Any sub-program can change a global, making bugs very hard to track. Use local variables and pass data through parameters instead.

✓ CHECK — Q10
A programmer writes a procedure that uses a variable called count declared inside it. After the procedure completes, the main program tries to print count. What happens?
