🛡️ Why Do Programs Need Defending?

Imagine you've built an app that asks users to enter their age. You expect a number like 15 or 42. But what happens when someone enters:

  • "fifteen" — text instead of a number
  • -50 — a negative age
  • 999 — an impossibly high age
  • "" — nothing at all

Without protection, your program might crash, produce incorrect results, or even be exploited by attackers.

Definition Defensive Design

A programming approach that anticipates how things might go wrong and includes measures to prevent or handle problems before they cause harm.

🏰 Think of it like building a castle...

You don't just build walls — you add a moat, guards at the gate, and backup supplies. Defensive design means building your program to withstand attack and misuse, not just to work in perfect conditions.

⚠️ Anticipating Misuse

Good programmers think like pessimists — they assume users will make mistakes, enter weird data, and sometimes even try to break things on purpose.

What Could Go Wrong?

When designing a program, you should consider:

🤦
Accidental Errors
Typos, wrong format, forgetting to fill in fields
User enters "O" instead of "0"
🤷
Misunderstanding
User doesn't know what format is expected
Date entered as "5/3/24" vs "05-03-2024"
😈
Deliberate Attack
Someone trying to crash or exploit the system
Entering code instead of data
🔄
Edge Cases
Extreme or boundary values that might break logic
Quantity of 0 items, or 999999999
💡 The Golden Rule

Never trust user input. Always assume data coming from outside your program could be wrong, malformed, or malicious — and handle it accordingly.

Real-World Example

A shopping website asks for quantity. Without defensive design:

quantity = input("How many items?") total = price * quantity # CRASH if quantity isn't a number!

If someone enters "five" instead of 5, the program crashes. If they enter -10, they might get money back instead of paying!

✅ What is Input Validation?

Definition Input Validation

The process of checking that data entered by a user meets specific criteria before the program accepts and processes it.

Validation acts as a gatekeeper — it checks incoming data and only allows through what meets your requirements. Bad data gets rejected with a helpful error message.

🚪 Nightclub Bouncer Analogy

Input validation is like a bouncer at a club. They check: Are you on the guest list? (presence) Are you old enough? (range) Is your ID valid? (format) Only if you pass all checks do you get in.

The Validation Process

# WITHOUT validation - dangerous! age = int(input("Enter age: ")) # Crashes on non-numeric input # WITH validation - safe! age_input = input("Enter age: ") if age_input.isdigit(): # Check it's a number age = int(age_input) if 0 <= age <= 120: # Check it's reasonable print("Age accepted") else: print("Age must be 0-120") else: print("Please enter a number")
⚠️ Validation ≠ Verification

Don't confuse these terms! Validation checks if data is reasonable/correctly formatted. Verification checks if data is accurate (e.g., double-entry of an email). Both are important but serve different purposes.

🔧 Validation Techniques

There are several common validation checks you should know. Each one catches a different type of problem:

📋
Presence Check
Ensures a field hasn't been left empty
✓ "John"
✗ "" (empty)
🔢
Type Check
Ensures data is the correct data type
✓ 25 (integer)
✗ "twenty-five"
📏
Range Check
Ensures a number falls within acceptable limits
✓ 45 (age 0-120)
✗ -5 or 200
📐
Length Check
Ensures text is within acceptable length
✓ "ABC123" (6 chars)
✗ "AB" (too short)
📝
Format Check
Ensures data matches a required pattern
✓ "AB12 3CD" (postcode)
✗ "12AB CD3"
📜
Lookup Check
Ensures value exists in a predefined list
✓ "Red" (from list)
✗ "Banana"
📝 Exam Tip

In exams, you may be asked to identify which validation type should be used for a given scenario, or to write pseudocode/code that implements a specific validation check. Make sure you can recognise AND implement each type.

📝 Notes for This Section

Validation Techniques

  • Presence check: Is there any data?
  • Type check: Is it the right data type?
  • Range check: Is the number within limits?
  • Length check: Is the text the right length?
  • Format check: Does it match the required pattern?
  • Lookup check: Is it in the allowed list?

🔐 Authentication

Definition Authentication

The process of confirming that a user is who they claim to be, typically before granting access to a system or data.

While validation checks if data is valid, authentication checks if the person is valid. It's about identity, not data format.

Common Authentication Methods

🔑
Something You Know
Password, PIN, security question
📱
Something You Have
Phone, security key, ID card
👆
Something You Are
Fingerprint, face scan, voice

Username & Password Authentication

The most common form of authentication. The system stores valid username/password combinations and checks if what the user enters matches:

# Simple authentication example STORED_USERNAME = "admin" STORED_PASSWORD = "secret123" entered_user = input("Username: ") entered_pass = input("Password: ") if entered_user == STORED_USERNAME and entered_pass == STORED_PASSWORD: print("Access granted!") else: print("Access denied.")
⚠️ Security Note

In real systems, passwords are never stored as plain text — they're encrypted (hashed). The example above is simplified for learning purposes. Also notice we don't tell the user which part was wrong — this prevents attackers from knowing if they got the username right.

🧩 Username Formula

Usernames are created by combining:
• First 2 characters of the employee's first name
• Last 2 characters of the employee's surname
• Last 2 digits of the year they joined
• All UPPERCASE

Example: Jennifer Dupington who joined in 2024 → JE + ON + 24 → JEON24

🔒

Secure Login

Employee: Samuel Richardson | Joined: 2019

💡 Password hint: It's one of the most commonly used passwords in the world, but follows password policy (starts with capital letter, includes lowercase, ends with a number, 8+ characters).

What Makes Good Authentication?

  • Strong password requirements — minimum length, mix of characters
  • Limited login attempts — lock account after too many failures
  • Vague error messages — don't reveal which part was wrong
  • Secure storage — never store passwords in plain text
  • Two-factor authentication (2FA) — combine two methods for extra security
