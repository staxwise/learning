🧠 The Brain Analogy

Imagine you're following a recipe to bake a cake. Your brain needs to keep track of several things at once:

🧠 Your Brain When Following a Recipe

"Which step am I on?" — You need to know you're on step 3, so you don't repeat step 2 or skip to step 5.

"Where do I need to look?" — You might need to check the cupboard at location B4 for flour.

"What am I currently holding?" — You're holding the ingredient you just grabbed, or the thing you're about to put down.

"What's my running total?" — If you're measuring, you need to keep track of accumulated amounts.

The CPU faces exactly the same challenges. It's following a "recipe" (a program), and it needs to track:

  • Which instruction it's currently on
  • Where to find/store data in memory
  • What data it's currently working with
  • The results of calculations
Key Term Register

A small, extremely fast storage location inside the CPU. Registers hold data that the CPU is actively working with. Think of them as the CPU's "working memory" — like the thoughts you hold in mind while completing a task.

🤔 Why Not Just Use RAM?

RAM is much slower than registers. If the CPU had to fetch every piece of data from RAM constantly, it would be like having to walk to a filing cabinet every time you needed to remember what step you're on. Registers are right there inside the CPU — instant access.

📍 Program Counter (PC)

🧠 Brain Analogy

"Which step am I on?"

When following a recipe, you keep mental track: "I've done steps 1 and 2, now I'm on step 3." Without this, you'd lose your place constantly.

Key Term Program Counter (PC)

A register that holds the memory address of the next instruction to be fetched. After each instruction is fetched, the PC automatically increments to point to the next one.

Program Counter (PC)
0042

↑ "The next instruction is at memory address 0042"

What the PC Does

  • Stores the address of the next instruction (not the current one)
  • Automatically increases by 1 after each fetch
  • Can be changed by jump/branch instructions (like skipping to step 10 in a recipe)
⚠️ Common Misconception

The PC doesn't hold the instruction itself — it holds the address where the instruction can be found. It's like knowing "step 3 is on page 5" rather than knowing what step 3 actually says.

📮 Memory Address Register (MAR)

🧠 Brain Analogy

"Where do I need to look?"

Before you can grab the flour, you need to know it's in cupboard B4. You hold that location in mind: "I need to go to B4." The MAR works the same way.

Key Term Memory Address Register (MAR)

A register that holds the memory address of the location the CPU wants to read from or write to. It's connected to the address bus and tells memory "this is the location I'm interested in."

Memory Address Register (MAR)
0042

↑ "I want to access memory location 0042"

What the MAR Does

  • Holds the address of data the CPU wants to read or write
  • Connected to the address bus
  • Used for both fetching instructions AND accessing data
🤔 MAR vs PC — What's the Difference?

Both hold addresses, so what's the difference?

PC: Keeps track of where we are in the program — the "next step" tracker.

MAR: The address we're actively sending to memory right now — "go to this shelf."

During fetch, the PC's value gets copied into the MAR. The MAR is what actually communicates with memory.

📦 Memory Data Register (MDR)

🧠 Brain Analogy

"What am I currently holding?"

When you grab the flour from cupboard B4, you're now holding it in your hands. You haven't used it yet — it's just in transit. Similarly, when putting something away, you hold it before placing it on the shelf.

Key Term Memory Data Register (MDR)

A register that holds data that has just been read from memory or data that is about to be written to memory. It acts as a buffer between the CPU and RAM. Also called the Memory Buffer Register (MBR).

Memory Data Register (MDR)
LDA 5

↑ "I'm currently holding the instruction 'LDA 5'"

What the MDR Does

  • Reading: Receives data from memory before the CPU uses it
  • Writing: Holds data the CPU wants to store before it goes to memory
  • Connected to the data bus
  • Two-way traffic: data flows in AND out through the MDR
⚠️ MDR vs MAR

MAR holds an address (where to look).

MDR holds the actual data (what was found there, or what we're putting there).

Think: MAR = "Shelf B4", MDR = "The flour that was on shelf B4"

  • Holds data read from memory OR data about to be written
  • Connected to the data bus
  • Like holding an item in your hands (in transit)
  • Two-way: handles both read and write operations
  • Also called Memory Buffer Register (MBR)

🧮 Accumulator (ACC)

🧠 Brain Analogy

"What's my running total / current result?"

When doing mental arithmetic like 5 + 3 + 7, you keep a running total in your head: "5... now 8... now 15." The accumulator is where the CPU keeps its "working answer."

Key Term Accumulator (ACC)

A register that stores the results of calculations performed by the ALU (Arithmetic Logic Unit). It's called the accumulator because results "accumulate" here — one calculation's result becomes the input for the next.

Accumulator (ACC)
00000015

↑ "My current calculation result is 15"

What the ACC Does

  • Stores results from ALU operations (add, subtract, compare, etc.)
  • Often used as one of the inputs for the next calculation
  • Central to the execute phase of the FDE cycle
  • Value can be stored back to memory when needed
🤔 Why "Accumulator"?

Imagine adding up a shopping list: 5 + 3 + 7 + 2. You don't calculate all at once — you accumulate:

ACC = 5 → ACC = 5+3 = 8 → ACC = 8+7 = 15 → ACC = 15+2 = 17

Each result stays in the accumulator, ready for the next operation.

🔄 The Fetch-Decode-Execute Cycle

Now we see how these registers work together. The CPU repeats this cycle billions of times per second:

FETCH → DECODE → EXECUTE → (repeat)
Key Term Fetch-Decode-Execute Cycle

The continuous cycle that the CPU follows to process every instruction. Fetch retrieves the instruction from memory, Decode interprets what it means, and Execute carries out the action.

The Three Phases

Fetch Get the instruction from memory

The PC tells us where to look. We copy that address to the MAR, retrieve the instruction, and it arrives in the MDR. Then we increment the PC.

Decode Work out what the instruction means

The Control Unit examines the instruction in the MDR and figures out what operation to perform and what data is needed.

Execute Carry out the instruction

The CPU performs the action — this might involve the ALU doing a calculation (result goes to ACC), or reading/writing more data using MAR and MDR.

🧠 Brain Analogy: Following a Recipe

Fetch: Read the next step from the recipe book

Decode: Understand what the step is asking ("add 100g flour")

Execute: Actually do it (measure and add the flour)

Then move to the next step and repeat!

📝 Notes for This Section

FDE Cycle Overview

  • Fetch: Get instruction from memory (uses PC, MAR, MDR)
  • Decode: Control Unit interprets the instruction
  • Execute: Carry out the operation (may use ACC, MAR, MDR)
  • Cycle repeats continuously, billions of times per second

📝 Complete Summary for Your Notes

The Four Registers

  • PC (Program Counter): Holds address of next instruction
  • MAR (Memory Address Register): Holds address being accessed → Address Bus
  • MDR (Memory Data Register): Holds data in transit → Data Bus
  • ACC (Accumulator): Holds results of ALU calculations

FDE Cycle

  • Fetch: PC → MAR → Memory → MDR → PC+1
  • Decode: Control Unit interprets instruction from MDR
  • Execute: Carry out operation (ALU results → ACC)

Key Connections

  • MAR connected to Address Bus
  • MDR connected to Data Bus
  • ACC connected to ALU
